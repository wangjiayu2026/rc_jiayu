## AI 使用说明

本项目在设计与实现过程中使用了 AI 辅助，但最终方案由本人结合课程目标、实现成本和个人工程判断确定。

### 1) AI 在哪些关键地方提供了帮助

- **技术选型建议**  
  AI 提供了 Go、Java、Python 等方案对比，并说明了各自在并发处理、工程复杂度、可维护性方面的差异。

- **代码实现与工程搭建**  
  AI 协助完成了项目骨架与核心代码实现，包括：
  - Java + Spring Boot 工程结构
  - SQLite 持久化队列（任务入队、状态流转、重试字段）
  - 异步 worker 投递与指数退避重试
  - 文档文件（README、DESIGN）初始化与整理

- **优化与后续演进建议**  
  AI 给出了后续可扩展方向，例如多实例竞争消费、队列化改造、供应商适配层、监控告警与治理能力等。

### 2) AI 给出过哪些没有采纳的建议

- **分布式一致性 / 更重型的 Exactly-once 方案**  
  AI 提到过更复杂的分布式一致性设计与更完整的企业级架构。  
  **未采纳原因**：当前作业以第一版可运行系统为目标，优先实现“至少一次 + 可重试”的核心能力，避免在 v0 过度设计。

- **多实例分布式架构（含重型队列中间件）**  
  AI 也给过引入更复杂分布式组件的方向。  
  **未采纳原因**：作业阶段优先单实例闭环，控制实现范围和调试复杂度。

### 3) 哪些关键决策由本人做出，以及原因

- **决策 1：选择 Java，不选择 Go**  
  **原因**：本人对 Java 更熟悉，能更稳定地完成实现、调试与答辩说明；Go 学习成本更高，会影响交付效率和质量。

- **决策 2：不使用 Kafka，使用 SQLite**  
  **原因**：作业 v0 的目标是可靠投递闭环，不是高吞吐分布式平台。SQLite 部署简单、依赖少、便于演示；Kafka 会显著增加环境与运维复杂度，属于过度设计。

- **决策 3：引入 `idempotency_key`，实现“至少一次”语义下的防重复入队**  
  **原因**：通知系统在网络抖动和重试场景下天然可能重复请求。通过 `idempotency_key` + 唯一索引，可以在入口侧降低重复入队风险，同时保持至少一次投递语义。

- **决策 4：不做健康检查接口（v0）**  
  **原因**：本次作业核心在可靠投递语义、失败处理、重试与状态管理。健康检查与完整监控告警属于运维治理能力，非本次核心目标，故在 v0 不纳入实现范围。

- **决策 5：加入崩溃恢复机制（回收卡住的 `DELIVERING` 任务）**  
  **原因**：这是在实现与验证过程中主动发现的潜在可靠性问题：实例可能在调用外部 API 后异常退出，导致任务卡在 `DELIVERING`。因此补充了超时回收并重置为 `RETRY` 的机制，确保恢复后可继续投递，符合“至少一次”语义。
