## 设计说明（v0 / 作业版）

### 1. 系统边界

- **本系统解决的问题**
  - **统一接入**：业务系统用统一的 HTTP API 提交“外部通知请求”（目标 URL、方法、Header、Body）。
  - **可靠投递**：请求先**落库持久化入队**，再由后台 worker 异步投递；失败自动重试并记录状态。
  - **可观测的最小闭环**：提供任务查询接口，能看到投递状态、重试次数、最后错误。

- **本系统明确不解决的问题（v0 不做的原因）**
  - **外部系统幂等/去重的最终正确性**：我们只能做到“至少一次”，无法保证外部只收到一次；外部需支持幂等或按业务键去重。v0 仅提供 `idempotency_key` 的简单防重复入队能力。
  - **复杂的供应商适配（协议/签名/模板/字段映射）**：v0 直接透传 method/url/headers/body；适配层属于下一阶段（需要配置中心与更强治理）。
  - **强运维能力（告警、Dashboard、灰度）**：v0 只保证能运行与可查询；生产需要监控/告警/审计链路。

### 2. 可靠性与失败处理

- **通知投递语义**
  - 选择 **至少一次（At-least-once）**：入队落库后一定会反复尝试，直到成功或进入死信。
  - 原因：外部 API 不可控，网络/超时/5xx 都可能导致“不确定成功”，为了不丢通知只能至少一次。

- **外部系统失败或长期不可用的策略**
  - **指数退避 + 抖动（jitter）重试**：避免在故障期间对外部形成雪崩压力。
  - **最大尝试次数**：超过 `NOTIFY_MAX_ATTEMPTS` 后进入 **dead-letter**（状态 `DEAD`），保留最后错误与重试次数，便于人工介入或后续补偿机制。
  - **超时控制**：每次投递有超时（默认 5s，可在请求里覆盖），避免 worker 被长期卡住。
  - **崩溃恢复**：如果实例在调用外部 API 后崩溃，任务可能停留在 `DELIVERING`；v0 增加“卡住回收”机制，把超过阈值的 `DELIVERING` 重置回 `RETRY`，因此恢复后仍会继续重试（可能导致外部收到重复通知，符合至少一次语义）。

### 3. 取舍与演进

- **哪些设计在 AI 的方案建议中可能“过度”，v0 不采纳**
  - **引入 Kafka / Pulsar / SQS**：对作业与 v0 规模过重；v0 用 SQLite 即可完成“持久化队列 + 重试”闭环。
  - **复杂的分布式一致性/Exactly-once**：成本极高且依赖外部配合；通知类场景通常以“至少一次 + 幂等”落地。
  - **健康检查端点/监控平台**：作业要求核心在“可靠投递语义与失败处理”，健康检查与完整监控通常需要运维体系（metrics、告警、Tracing），因此在 v0 先不实现。

- **未来流量或复杂度增长时如何演进**
  - **存储与队列**：从 SQLite 升级到 Postgres/MySQL（行级锁、多实例）或外部队列（Kafka/RabbitMQ）以支撑水平扩展。
  - **多实例与竞争消费**：引入“租约/锁”或队列系统的 consumer group，实现多副本 worker 并发处理。
  - **治理能力**：按租户限流、并发上限、熔断、隔离（不同供应商不同队列/权重）。
  - **可观测性**：指标（成功率/延迟/重试分布）、告警（dead-letter 激增、外部 5xx 激增）、审计与追踪。
